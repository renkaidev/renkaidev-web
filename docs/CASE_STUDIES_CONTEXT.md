# CASE_STUDIES_CONTEXT.md

# Purpose

This document provides context about the real projects completed by RenkaiDev.

Its purpose is NOT to generate a portfolio or résumé.

Its purpose is to help AI assistants create and maintain the "Case Studies" section of the RenkaiDev services website.

Every case study should demonstrate problem solving, business understanding and technical execution instead of simply listing technologies.

---

# Writing Guidelines

Each case study should follow this structure:

1. Business Context
2. Client Problem
3. Analysis Process
4. Proposed Solution
5. Technical Implementation
6. Results
7. Technologies Used
8. Lessons Learned

Never exaggerate results.

Never invent metrics.

Never fabricate clients.

Never fabricate technologies.

If information is unknown, explicitly indicate that instead of guessing.

---

# RenkaiDev Engineering Philosophy

Projects begin by understanding the business, not by writing code.

The process normally follows:

Business Observation
↓
Problem Identification
↓
Requirements Gathering
↓
Documentation
↓
Solution Design
↓
Architecture
↓
Development
↓
Deployment
↓
Continuous Improvement

Technology is only a tool.

The objective is always solving business problems.

# Case Study 01

## Del Valle Construcciones

Status:
In Development

Year:
2026

Role:
End-to-end developer.

Project Type:
Commercial website + lead acquisition system + reference quotation simulator.

Client Problem

The company had almost no digital presence.

Its commercial strategy depended primarily on word-of-mouth referrals and a personal Facebook profile instead of a business page.

There was no structured lead acquisition process.

There was no digital marketing strategy.

Customer requests had no traceability.

The quotation process depended entirely on manual work.

Analysis

The project started with observation of the company's operation.

Requirements were gathered through:

- meetings
- interviews
- WhatsApp
- emails
- Zoom meetings
- Google Forms
- documentation
- site visits

The analysis also identified future opportunities beyond the initial scope, including:

- inventory management
- logistics
- operational traceability
- expense tracking

These were documented for future phases.

Solution

A modern web platform was proposed including:

- corporate website
- lead capture
- reference quotation simulator
- customer database
- commercial follow-up
- better professional image

The quotation simulator is not intended to replace technical site inspections.

Its purpose is to structure the commercial process and generate traceable opportunities.

Architecture

Frontend:
Next.js

Backend:
FastAPI

Database:
PostgreSQL

Infrastructure:
Linux VPS

Cloudflare

Architecture:
Separated Frontend + REST API

Principles

MVC

SOLID

DRY

KISS

Clean Code

Clean Architecture

Lessons Learned

The most important lesson was that understanding business processes generates more value than simply building software.

# Case Study 02

## Intelligent Agenda Automation

Status:
MVP

Year:
2026

Problem

The director and secretary maintained separate agendas.

Operational redundancy caused inconsistencies and information loss.

Solution

A workflow was designed using n8n to automate synchronization between both agendas.

Implementation stopped before production because the required infrastructure was unavailable.

Lessons Learned

Even relatively small automations can significantly improve internal coordination.

Infrastructure constraints can determine project viability.
